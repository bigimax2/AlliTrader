## Логика работы

### Проверка токенов
Приложение проверяет, есть ли у пользователя хотя бы один токен с правами `esi-assets.read_assets.v1`.

### Получение персонажей
Все персонажи пользователя, у которых есть токен доступа к ассетам, получаются через `OwnershipRecord`:

```python
characters_with_assets = EveCharacter.objects.filter(
    ownership_records__user=request.user
).order_by('name').distinct()
```

### Фильтрация
Фильтры применяются по следующим полям:
- Персонаж
- Локация (станция)
- Тип локации (Asset Safety, Hangar, Cargo и т.д.)
- Тип актива (распакован/не распакован)
- Категория предмета
- Группа предмета

### Представление данных
Все ассеты отображаются в одной таблице со следующими колонками:
1. Персонаж
2. Локация
3. Тип локации (badge)
4. Предмет
5. Категория
6. Группа
7. Количество
8. Тип актива (badge: Распакован/Не распакован)

## Технические детали

### Модель Asset
Используется существующая модель `trader.models.Asset` со связями:
- `character` → `EveCharacter`
- `location` → `EveLocation`
- `type_id` → `EveItemType`

### Оптимизация
Для оптимизации запросов используется `select_related` для всех связей:
```python
assets = assets.select_related('character', 'type_id', 'location')
```

### Производительность
- Для больших наборов данных рекомендуется использовать фильтрацию
- Таблица адаптивна и работает с большими объёмами данных
- Поиск по названию локаций работает на клиенте (JavaScript)
